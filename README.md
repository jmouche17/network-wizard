# üßô Network Wizard

**Network Wizard** is a self-hosted web-based network automation platform built for network engineers. It provides a clean browser interface for managing devices, running scripts, chaining automations into playbooks, and monitoring your infrastructure ‚Äî without requiring any client software beyond a web browser.

---

## Features

- **Device inventory** ‚Äî store devices with encrypted credentials, groups, tags, and up to 10 custom fields
- **Script runner** ‚Äî execute Python or Bash scripts against one or many devices simultaneously with real-time streaming output
- **Playbooks** ‚Äî chain scripts into ordered sequences with per-step inputs, delays, and stop-on-failure logic
- **Scheduler** ‚Äî run scripts or playbooks on a cron schedule automatically
- **SSH terminal** ‚Äî interactive browser-based SSH sessions to any device
- **Telnet terminal** ‚Äî raw TCP terminal for legacy devices
- **Network tools** ‚Äî Ping, Traceroute, DNS lookup, Port scanner, TFTP server
- **Config backups** ‚Äî save and diff device configurations with color-coded comparison
- **File manager** ‚Äî upload, create, and edit template files used by scripts
- **Webhook system** ‚Äî send notifications to Slack/Teams/custom endpoints and receive triggers from external systems
- **MFA support** ‚Äî per-user TOTP two-factor authentication (Google Authenticator, Authy, 1Password)
- **Multi-user** ‚Äî role-based access with admin and operator roles

## Quick Start

### Prerequisites

- [Docker Desktop](https://www.docker.com/products/docker-desktop/) (Mac / Windows / Linux)
- Git or a way to download the release zip

### Installation

**1. Download and extract**

```bash
# download the latest release zip and extract it
unzip network-wizard.zip
cd device-runner
```

**2. Run the setup script**

```bash
bash docker-setup.sh
```

This will:
- Verify Docker is installed
- Generate secure random `SECRET_KEY` and `ENC_KEY` values
- Create a `.env` file with your secrets
- Build and start the container

**3. Open the app**

Navigate to `http://localhost:5000` in your browser.

Default credentials: `admin` / `admin`

> **Important:** Change the default password immediately after first login under the Users tab.

## Updating

```bash
# copy new files into the project folder, then rebuild
docker compose down
docker compose up --build -d
```

Your data in `data/`, `scripts/`, `uploads/`, and `backups/` is stored in mounted volumes and is never affected by rebuilds.

## ‚öÝÔ∏è Important ‚Äî Protect your `.env` file

The `.env` file contains the `ENC_KEY` used to encrypt all stored device passwords and API tokens. 

- **Never commit `.env` to Git** ‚Äî it is in `.gitignore` by default
- **Never delete or regenerate `.env`** once you have real device data ‚Äî you will not be able to decrypt existing credentials
- Back it up somewhere secure (password manager, vault, etc.)

## Configuration

| Variable | Description |
|---|---|
| `SECRET_KEY` | Flask session signing key ‚Äî random 64-char hex |
| `ENC_KEY` | Fernet AES encryption key for stored credentials |

Both are auto-generated by `docker-setup.sh`. Port can be changed by editing `docker-compose.yml`:

## Use Cases

### 1. Multi-site Fortigate Configuration Deployment

Managing dozens of Fortigate firewalls across multiple sites with consistent baseline configs is time-consuming and error-prone when done manually. Network Wizard lets you define a template config file once and push it to every device in a group with a single click.

**How it works:**

1. Add all Fortigate devices to Network Wizard with their API tokens stored encrypted
2. Create a template file (`fortigate_vlan_config.txt`) in the Files tab with placeholders:

```
config system interface
edit VL.100
set alias DATA-LAN
set type vlan
set vlanid 100
set ip {{vlan_100}}
set interface fortilink
next
end
```

3. Write a script that reads the template, substitutes per-site IP values collected from the input form, and pushes via Netmiko or the Fortigate REST API
4. Use the `# INPUTS:` block to let the engineer fill in site-specific values (VLAN IPs, subnet masks) before running
5. Run against the "Fortigate" device group ‚Äî all devices get configured in parallel with real-time output streaming to the console

**Result:** What used to take 2-3 hours of SSH copy-paste across 15 firewalls takes 5 minutes.

---

### 2. Automated Daily Config Backups with Drift Detection

Unauthorized config changes are one of the hardest problems to catch without dedicated tooling. Network Wizard's scheduler and backup viewer let you capture daily snapshots and immediately see what changed.

**How it works:**

1. Write a backup script using Netmiko's `send_command("show running-config")` that saves output to the `backups/` folder with a timestamp in the filename:

```python
from netmiko import ConnectHandler
from datetime import datetime

conn = ConnectHandler(device_type="cisco_ios", host=device["ip"],
    username=device["username"], password=device["password"])

config    = conn.send_command("show running-config")
timestamp = datetime.now().strftime("%Y%m%d_%H%M%S")
filename  = f"backups/{device['name']}_backup_{timestamp}.cfg"

with open(filename, "w") as f:
    f.write(config)

print(f"Saved {filename}")
conn.disconnect()
```

2. Schedule it to run nightly at 2am against all devices using the Scheduler tab (`0 2 * * *`)
3. When an incident occurs, open the Backups tab, select yesterday's and today's backup for the affected device, and click **Compare** ‚Äî a color-coded diff shows exactly what changed and when

**Result:** Instant visibility into unauthorized or accidental changes. Pairs well with an outbound webhook to Slack that fires when the diff is non-empty.

---

### 3. New Site Onboarding Playbook

Standing up a new branch office requires the same sequence of steps every time ‚Äî configure VLANs, set up routing, push security policies, verify connectivity. A Network Wizard playbook captures this as a repeatable, auditable process.

**How it works:**

1. Add the new site's devices to Network Wizard with their credentials and site-specific UDF values (e.g. UDF1 = site code, UDF2 = ISP circuit ID)
2. Build a playbook with steps like:
   - Step 1: `push_vlan_config.py` ‚Äî configures VLAN interfaces (reads from template file)
   - Step 2: `configure_routing.py` ‚Äî sets up BGP or static routes
   - Step 3: `push_security_policies.py` ‚Äî applies firewall rules
   - Step 4: `verify_connectivity.py` ‚Äî pings key destinations and reports pass/fail
3. Enable **Stop on failure** so a broken VLAN config doesn't cascade into a routing push on top of a broken foundation
4. Run the playbook against the new site's device group ‚Äî each step's output streams live to the console

**Result:** A new site goes from bare metal to fully configured in a documented, repeatable sequence. The run history logs which engineer ran it, when, and what the output was.

## Security Notes

- All device passwords and API tokens are encrypted at rest using Fernet (AES-128-CBC)
- User passwords are hashed with SHA-256 + salt ‚Äî never stored in plaintext  
- Enable TOTP MFA for all admin accounts under the Users tab
- Scripts run in a sandboxed subprocess ‚Äî they cannot access files outside the app directory
- Rate limiting is applied to the login endpoint (10 attempts/minute)
- In production, run behind nginx with HTTPS ‚Äî see `deploy/` folder for config

## Script Reference

Scripts have access to these injected variables:

```python
device['name']          # device name
device['ip']            # IP address
device['port']          # port number
device['username']      # username
device['password']      # decrypted password
device['api_token']     # decrypted API token
device['protocol']      # SSH / HTTPS / Telnet / REST
device['group']         # group name
device['tags']          # list of tags
device['udf1']          # user defined field 1 (up to udf10)

devices                 # dict of all devices
env_vars                # dict of environment variables
script_inputs           # dict of user-supplied inputs (from # INPUTS: block)
files                   # file helper: files.read(), files.lines(), files.exists()
selected_file           # full path to file chosen in the UI picker
selected_filename       # just the filename
```

### Declaring script inputs

```python
# INPUTS:
# hostname:  text                      | Target hostname    | switch.corp.local
# vlan_id:   number                    | VLAN ID            | 100
# password:  password                  | Enable password
# dry_run:   checkbox                  | Dry run only       | false
# protocol:  select[SSH,HTTPS,Telnet]  | Protocol           | SSH
```
